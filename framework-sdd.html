🏗️架構與工具研發：SDD 自動化測試框架

為解決測試腳本維護成本高、需求變動與測試案例脫節的痛點，研發並搭建一套基於 Spec-Driven Development (SDD) 核心思想的 E2E 測試架構。

💡專案開發背景

在傳統流程中，開發文檔、測試案例與自動化腳本往往是孤立的。透過 SDD (規格驅動開發)，我們將測試規格書作為「唯一真相來源 (Single Source of Truth)」，實現從需求定義到自動化執行的無縫銜接。

技術棧與研發工具

層級 選用工具 / 技術 研發價值
核心框架 Playwright + .NET / C# 利用 Playwright 的 Auto-wait 與隔離性，解決 Selenium 長期以來的 Flaky Tests 問題。
規格定義 SpecFlow (Gherkin syntax) 使用自然語言描述測試場景，非技術人員亦能校對測試邏輯。
開發模式 Page Object Model (POM) + SDD 將 UI 元素與測試邏輯分離，當規格異動時，僅需維護對應的 Step Definitions。
AI 輔助 LLM Agent Integration 研發 AI 代理工具，根據 API 文件自動生成基礎 NUnit 測試代碼與 Mock Data。

🚀架構設計亮點

🛠️ 模組化與可擴展性

  • 自主研發跨資料庫處理工具，支援 Oracle 與 SQL Server 資料一致性校驗。
  • 封裝 ECPay 支付模組，實現一行代碼即可呼叫完整的金流支付模擬。

📊 自動化報告與整合

  • 自定義 GitLab CI/CD Pipeline 流程。
  • 串接 Microsoft Teams Webhook，測試失敗時自動發送即時警示與 Base64 截圖報告。

SDD 研發流程架構

[規格書定義] ➔ [Step Definitions 自動產生] ➔ [Playwright 邏輯實作] ➔ [CI/CD 自動執行]

📈研發效益

  • 降低維護成本：腳本重用率提升 40%，需求變動後的修復時間減少 50%。
  • 工程效率提升：透過 AI 工具輔助生成測試案例，縮短 30% 的專案初期搭建時間。
  • 團隊協作優化：Kuan-lin 等工程團隊可透過直觀的 Spec 報告快速理解測試覆蓋點，減少溝通誤差。
